# Prompt v6 변경 내역

## 개요

v6은 v5의 `schema_linking.j2`를 개선하여 **스키마 링킹 출력 구조를 확장**한 버전입니다.
`select_all`, `limit` 필드를 새로 추가하고, `column_mappings` 제외 규칙을 구체화했습니다.
`generate_sql.j2`도 새 필드를 처리하도록 함께 수정했습니다.

---

## 파이프라인 비교

```
v5 (5단계):
  Step1 IR → Step2 스키마링킹(LLM) → Step3 Rewrite(LLM) → Step4 SQL(LLM) → Step5 DB실행

v6 (5단계):
  Step1 IR → Step2 스키마링킹(LLM) → Step3 Rewrite(LLM) → Step4 SQL(LLM) → Step5 DB실행
```

파이프라인 단계 수는 동일합니다.

---

## 각 프롬프트 역할

| 파일 | 역할 | v5 대비 |
|---|---|---|
| `schema_linking.j2` | 원본 질문 → 테이블·컬럼 매핑 | **수정** (아래 참고) |
| `rewrite_query.j2` | 스키마 링킹 결과 기반 질문 재작성 | **변경 없음** |
| `generate_sql.j2` | SQL 생성 | **수정** (아래 참고) |

---

## 핵심 변경: schema_linking.j2

### 추가된 출력 필드

**1. `select_all` (신규)**

질문이 "전체 정보", "모든 컬럼", "전부 보여줘" 등의 의미일 경우 `true`.
`select_all: true`이면 `column_mappings`는 반드시 빈 객체 `{}`로 둔다.
→ SQL 생성기가 `SELECT *` 를 명확히 판단할 수 있도록 구조화.

**2. `limit` (신규)**

"상위 N개", "N건", "최근 N건" 등 개수 제한 의도를 정수로 추출.
→ SQL 생성기가 `LIMIT {N}` 을 정확히 생성할 수 있도록 구조화.

### column_mappings 제외 규칙 구체화

v5까지는 `column_mappings`에 집계·정렬·시간 표현이 혼입될 수 있었습니다.
v6에서는 포함 금지 목록을 명시적으로 정의했습니다:

- `*` (전체 컬럼)
- 집계함수 포함 표현 (예: COUNT(id))
- 정렬 표현 ("많은 순", "가장 최근", "오래된 순")
- 개수 제한 표현 ("상위 N건", "N개")
- 시간 필터 표현 ("어제", "지난주")
- "전체 정보", "모든 컬럼", "전부"

### time_column 판단 기준 명확화

"가장 최근" 은 시간 필터가 아닌 정렬 의도로 구분.
→ `time_column`이 아닌 `sort_order: DESC` 로 처리.

---

## 핵심 변경: generate_sql.j2

**1. `select_all` 처리 규칙 추가**

```
- `select_all` → true이면 SELECT * 사용, column_mappings 무시
```

**2. `limit` 처리 규칙 추가 (LIMIT 우선순위 정의)**

```
- limit 필드가 있으면 → LIMIT {limit값} 사용
- limit가 null이고 COUNT가 아니면 → LIMIT 200 사용 (기본값)
- COUNT 집계는 LIMIT 불필요
```

v5까지는 `LIMIT 200`이 항상 적용되어 사용자가 "상위 5개"라고 해도 무시되었습니다.

## 평가 결과 (v6)

> v5 대비 전체 정답률이 하락하였습니다. `schema_linking.j2`의 규칙 복잡도 증가로
> 스키마 링킹 단계에서 오류가 늘어난 것으로 판단됩니다.

### 난이도별 정답률

| 난이도 | 정답 | 전체 | 정답률 |
|--------|------|------|--------|
| 1-1 (단순) | 12 | 19 | **63.2%** |
| 1-2 (중간) | 5 | 19 | **26.3%** |
| 1-3 (복잡) | 1 | 11 | **9.1%** |
| **전체** | **18** | **49** | **36.7%** |

### 오류 타입별 분포

| 오류 타입 | 건수 | 비고 |
|-----------|------|------|
| WRONG_COLUMN | 17건 | 가장 많음 — 컬럼 매핑 오류 |
| WRONG_CONDITION | 6건 | WHERE 조건 오류 |
| WRONG_AGGREGATION | 3건 | 집계 함수 오류 |
| TYPE_MISMATCH | 2건 | 타입 불일치 |
| WRONG_TABLE | 2건 | 테이블 선택 오류 |
| NO_SQL | 1건 | SQL 미생성 |

### 단계별 평균 소요 시간

| 단계 | 평균 | 최소 | 최대 |
|------|------|------|------|
| 스키마 검색 | 0.23s | 0.13s | 0.32s |
| 스키마 링킹 | **25.18s** | 13.68s | 307.44s | 
| 질의 재작성 | 6.91s | 3.40s | 15.88s |
| SQL 생성 | 5.02s | 1.85s | 12.79s |
| DB 실행 | 0.08s | 0.08s | 0.10s |
| **전체 합계** | **44.67s** | 21.07s | 383.59s |

> 스키마 링킹 평균 25.18s, 최대 307.44s로 병목이 심각합니다.
> 규칙이 많아진 프롬프트로 인해 LLM 추론 시간이 증가한 것으로 보입니다.

## v5 대비 평가

| | v5 | v6 |
|--|----|----|
| 전체 정답률 | v5 결과 참고 | 36.7% ↓ |
| WRONG_COLUMN | — | 17건 (최다 오류) |
| 스키마 링킹 평균 시간 | — | 25.18s (병목) |
| select_all 처리 | 미지원 | 지원 |
| limit 구조화 | 미지원 | 지원 |
| column_mappings 노이즈 | 발생 가능 | 제외 규칙 명시 |

### 퇴보 원인 분석

1. **규칙 복잡도 증가** — 출력 필드가 늘어나고 제외 규칙이 많아지면서 SLM 모델이 지시를 온전히 따르지 못하는 경우 증가
2. **WRONG_COLUMN 17건** — `column_mappings` 제외 규칙이 오히려 필요한 컬럼을 누락시키는 부작용 발생 가능성
3. **스키마 링킹 지연** — 프롬프트 복잡도에 따른 추론 시간 증가 (avg 25s, max 307s)
